FAQ

Suggest a Frontend

Requirements

  1. Open Source
  2. Enhances Privacy over main site
  3. Can be self-hosted. So, It's Decentralized
  4. Using it is really hard without automated redirections

Good Examples

  • Nitter
  • Invidious
  • Libreddit
  • SimplyTranslate

Bad Examples

  • PixivMoe - It wasn't meant to be an alternative private frontend in the first place
  • Startpage - It doesn't require automated redirections. Just set it as your default Search Engine. Plus, there's already an extension for it.
  • Old Reddit - It attempts to track/fingerprint you. Plus, there's already an extension for it.
  • Pinafore - It's a full featured web app for Mastodon, not a frontend
  • Soju - It's just a website that gives you samples of Spotify songs, not a frontend. If you want to use it, just visit their website and paste the spotify link you want to get a sample of.
  • Imginn - Has so many trackers. Wasn't meant to be an alternative private frontend in the first place.
  • PornInvidious: See Discussion

Why forking Privacy Redirect?

Privacy Redirect hasn't been maintained since 6/12/2021. Many instances are hard coded. Some should be removed as they became offline and many others should be added. Many Pull Requests were left pending and waiting. So we stepped in. Later, we started improving it further by adding new features and frontends.

Disclaimer: Even though we forked from Privacy Redirect, LibRedirect is ahead of the main project with several improvements and new features. LibRedirect is independent now.

Why all those permissions?

Let's examine them one by one:

  • webRequest, webRequestBlocking: To block http requests or redirect them. For example, to redirect to Invidious before it can reach YouTube's servers.
  • storage: To save LibRedirect's settings permanently.
  • Clipboard.writeText: To write the original link to the clipboard while using Copy Raw. This permission is used to only write data. LibRedirect does NOT read the data in your clipboard.
  • menus: For adding some buttons to the Context menues:

  • all_urls: LibRedirect is dynamic and customizable. The targets e.g youtube.com, twitter.com are written in Regex and are too complicated to be hard-coded in the manifest. Ex: . Furthermore, we also need to access the instances sites to inject some localStorage variables.

Where the hell are those instances coming from?

Almost every frontend project has a list of public instances. Examples: Invidious' official instances list, Nitter's official instances list. We fetch those lists using an automated python script. This script gets triggered every 3 days. We don't intercept nor modify the lists outside of removing urls that are not valid (the structure is not that of a url). They're not our responsibility, they're the projects maintainers'.

Chrome Web Store?

We can't publish it to the Chrome Web Store as it requires Manifest v3 and we use Manifest v2. You can still install it manually. We are planning to adopt Manifest v3 in Firefox as they will support the feature LibRedirect needs. If other browsers support Mozilla's policy, you might see LibRedirect in the Chrome Web Store.
